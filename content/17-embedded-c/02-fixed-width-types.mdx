Here's a question most C tutorials skip: **how big is an `int`?** The honest answer is "it depends." On your desktop, it's almost certainly 4 bytes. On a 16-bit microcontroller like the MSP430, it's 2 bytes. On some DSP architectures, it might be something else entirely.

C deliberately leaves the sizes of `int`, `long`, and `short` up to the platform. This was a brilliantly practical decision in 1972 — it let C run efficiently on wildly different hardware by using each CPU's natural word size. But it creates a serious problem: code that works perfectly on your development machine can produce wrong results, buffer overflows, or protocol violations when compiled for a different target.

## Why Variable Sizes Matter

Consider a network protocol that says "the message length is a 4-byte unsigned integer." If you write `unsigned int length;` and your code runs on a platform where `int` is 2 bytes, you can only represent lengths up to 65,535 instead of 4,294,967,295. Your program silently truncates large messages. Worse, if you write the struct to the network with `send(sock, &header, sizeof(header), 0)`, you send 2 bytes instead of 4, and the receiver reads garbage.

The same problem appears in file formats, hardware register access, and any code that needs to agree with the outside world on the exact layout of data in memory.

## stdint.h: The Solution

C99 introduced `<stdint.h>`, which provides types with **guaranteed exact widths**. `uint32_t` is always exactly 32 bits, everywhere. `int8_t` is always exactly 8 bits. No ambiguity, no platform dependence.

The naming convention is simple: `[u]int<N>_t` where `u` means unsigned and `N` is the bit count. Use these anywhere the size matters: protocol headers, file formats, hardware registers, and cross-platform data structures.

<Editor defaultCode={`#include <stdio.h>
#include <stdint.h>  // C99 fixed-width types
#include <inttypes.h> // Format macros for printing

int main() {
    printf("=== Fixed-Width Integer Types ===\\n\\n");

    // Exact width: guaranteed to be exactly N bits
    int8_t   a = 127;       // Exactly 8 bits, signed
    uint8_t  b = 255;       // Exactly 8 bits, unsigned
    int16_t  c = 32767;     // Exactly 16 bits, signed
    uint16_t d = 65535;     // Exactly 16 bits, unsigned
    int32_t  e = 2147483647;// Exactly 32 bits, signed

## When to Use What: The Decision Guide

Not every variable needs a fixed-width type. Here's the mental model:

- **`int`**: Use for general-purpose computation — loop counters, local calculations, function return values. The compiler picks the most efficient size for the platform, and that's what you want for speed.
- **`uint8_t`, `uint16_t`, `uint32_t`, `uint64_t`**: Use when the **exact size matters** — talking to hardware, sending data over a network, writing to a file, or defining a data structure that must have a specific layout.
- **`size_t`**: Use for anything that represents a size in bytes or an array index. It's unsigned and matches the pointer width of the platform, so it can index the largest possible array. `strlen()` returns `size_t`, `malloc()` takes `size_t`, and `sizeof` evaluates to `size_t`.
- **`int_fast32_t`** and friends: Use when you need *at least* N bits but want the CPU's fastest type at that width or larger. On a 64-bit machine, `int_fast16_t` might be 64 bits because 64-bit operations are the CPU's natural speed.

## The fast and least Variants

`<stdint.h>` provides two additional families beyond the exact-width types, each serving a different need:

**`int_leastN_t`**: Guarantees at least N bits in the smallest possible type. Use this when you're optimizing for memory — perhaps you have an array of millions of elements and want the smallest type that holds your value range.

**`int_fastN_t`**: Guarantees at least N bits in the fastest type the CPU can work with. On many architectures, operating on a 16-bit value requires an extra instruction to mask or sign-extend — an `int_fast16_t` avoids this by using a wider type when it's faster.

<Editor defaultCode={`#include <stdio.h>
#include <stdint.h>

int main() {

## Byte Order (Endianness)

When a multi-byte integer sits in memory, which byte comes first — the most significant or the least significant? This is called **endianness**, and it's one of those things that doesn't matter until it matters a lot.

Your x86 desktop stores the least significant byte first (**little-endian**): the number `0x12345678` sits in memory as `78 56 34 12`. Network protocols, by convention, use big-endian (most significant byte first): `12 34 56 78`. If you send bytes from memory directly onto the network without converting, the receiver interprets `0x78563412` — a completely different number.

This is the reason `htonl()`/`ntohl()` exist: they convert between host and network byte order. On a big-endian machine, they're no-ops. On a little-endian machine, they reverse the bytes. Always use them (or equivalent functions) when crossing byte-order boundaries.

<Editor defaultCode={`#include <stdio.h>
#include <stdint.h>
#include <string.h>

// Byte order matters for network protocols and file formats

int is_little_endian(void) {
    uint16_t x = 1;
    return *(uint8_t*)&x == 1;
}

// Convert between host and network byte order
uint16_t my_htons(uint16_t h) {
    uint8_t bytes[2];
    bytes[0] = (h >> 8) & 0xFF;  // High byte first (big-endian)
    bytes[1] = h & 0xFF;
    uint16_t result;
    memcpy(&result, bytes, 2);
    return result;
}

uint32_t my_htonl(uint32_t h) {
    uint8_t bytes[4];
    bytes[0] = (h >> 24) & 0xFF;
    bytes[1] = (h >> 16) & 0xFF;
    bytes[2] = (h >> 8) & 0xFF;
    bytes[3] = h & 0xFF;
    uint32_t result;
    memcpy(&result, bytes, 4);
    return result;
}

int main() {
    printf("=== Byte Order (Endianness) ===\\n\\n");

    printf("This system is: %s-endian\\n\\n",
           is_little_endian() ? "little" : "big");

    // Show how 0x12345678 is stored in memory
    uint32_t val = 0x12345678;
    uint8_t *bytes = (uint8_t*)&val;

    printf("Value: 0x%08X\\n", val);
    printf("Bytes in memory: ");
    for (int i = 0; i < 4; i++) printf("%02X ", bytes[i]);
    printf("\\n\\n");

    printf("Little-endian: 78 56 34 12 (LSB first)\\n");
    printf("Big-endian:    12 34 56 78 (MSB first)\\n\\n");

    printf("Network byte order is big-endian.\\n");
    printf("Use htons/htonl when sending, ntohs/ntohl when receiving.\\n");

    return 0;
}
`} />

<Callout type="info">
When writing data to files or networks, always specify the byte order. Use `htonl`/`htons` (host to network) and `ntohl`/`ntohs` (network to host) from `<arpa/inet.h>`, or write your own byte-swapping functions for portability. The safest approach for serialization is to always shift-and-mask into a byte array — this is endian-neutral and works everywhere.
</Callout>
