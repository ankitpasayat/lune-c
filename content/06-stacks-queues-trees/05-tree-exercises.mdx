These exercises build on the tree concepts from the previous lessons. Each problem tests a different aspect of recursive tree thinking. Before diving in, let's review the input format and build intuition for each technique.

## Tree Input Format

For all exercises, trees are given as: N (number of nodes, 0-indexed), then N lines each containing the value, left child index, and right child index (-1 means no child). Node 0 is always the root.

Example for this tree:
```
       10
      /  \
     5    15
    / \
   3   7
```
Input:
```
5
10 1 2
5 3 4
15 -1 -1
3 -1 -1
7 -1 -1
```

Node 0 has value 10, left child is node 1 (value 5), right child is node 2 (value 15). Node 1 has children at indices 3 and 4. Nodes 2, 3, and 4 are leaves (-1 means no child).

## Tree Height: The `max(left, right) + 1` Intuition

Computing a tree's height is one of the purest examples of recursive thinking. The key question: **how tall is a tree?** It's as tall as its tallest subtree, plus one for the root.

Picture it physically: if the left subtree is 3 levels deep and the right subtree is 5 levels deep, the whole tree is 6 levels deep (5 from the taller subtree + 1 for the root connecting to it).

```
height(node) = 1 + max(height(left), height(right))
height(NULL) = -1    (an empty tree has no edges)
```

Why -1 for the base case? Because height counts *edges*, not nodes. A single node (a leaf) has height 0 — zero edges below it. Its children are both NULL, and `1 + max(-1, -1)` = `1 + (-1)` = 0. The math works out cleanly.

Let's trace through the tree above:
- `height(3)` = `1 + max(-1, -1)` = 0 (leaf)
- `height(7)` = `1 + max(-1, -1)` = 0 (leaf)
- `height(15)` = `1 + max(-1, -1)` = 0 (leaf)
- `height(5)` = `1 + max(height(3), height(7))` = `1 + max(0, 0)` = 1
- `height(10)` = `1 + max(height(5), height(15))` = `1 + max(1, 0)` = 2

The function doesn't "look" at the whole tree at once. It asks each subtree "how tall are you?" and combines the answers. This is the essence of recursive tree thinking.

<Editor defaultCode={`#include <stdio.h>

int val[100], left_c[100], right_c[100];

int height(int node) {
    if (node == -1) return -1;
    int lh = height(left_c[node]);
    int rh = height(right_c[node]);
    return 1 + (lh > rh ? lh : rh);
}

int main() {
    int n;
    scanf("%d", &n);
    for (int i = 0; i < n; i++) {
        scanf("%d %d %d", &val[i], &left_c[i], &right_c[i]);
    }
    printf("Height: %d\\n", height(0));
    return 0;
}
`} time="O(n)" space="O(h)" />

## Counting Nodes at Each Level

This problem introduces a different recursive pattern: passing **extra information down** from parent to child. Instead of combining results going *up* (like height), you pass the current level *down* as a parameter.

Think of it as each node asking: "What level am I on?" The root is level 0. Its children are level 1. Their children are level 2. Each recursive call increments the level by 1, and the node records itself in the appropriate bucket.

<Exercise
  title="Nodes per level"
  description="Read a tree. Print the number of nodes at each level, from level 0 (root) downward, one count per line."
  starterCode={`#include <stdio.h>

int val[100], lc[100], rc[100];
int level_count[100];
int max_level;

void count_levels(int node, int level) {
    if (node == -1) return;
    level_count[level]++;
    if (level > max_level) max_level = level;
    // recurse
}

int main() {
    int n;
    scanf("%d", &n);
    for (int i = 0; i < n; i++) {
        scanf("%d %d %d", &val[i], &lc[i], &rc[i]);
    }

    max_level = 0;
    for (int i = 0; i < 100; i++) level_count[i] = 0;

    count_levels(0, 0);

    for (int i = 0; i <= max_level; i++) {
        printf("%d\\n", level_count[i]);
    }
    return 0;
}
`}

## Mirroring a Tree: Swap Children at Every Level

Mirroring a tree means flipping it horizontally — the left subtree becomes the right and vice versa. The result is what you'd see if you held the tree up to a mirror.

The algorithm is deceptively simple: **at every node, swap its left and right children.** That's it. But think about *why* this works: when you swap the children of the root, the entire left subtree moves to the right side and vice versa. But each subtree is still un-mirrored internally. So you recurse into both subtrees and swap their children too — and their children's children, all the way down to the leaves.

```
Original:           Mirrored:
      10                10
     /  \              /  \
    5    15           15    5
   / \                    / \
  3   7                  7   3
```

The in-order of the original is `3 5 7 10 15`. The in-order of the mirror is `15 10 7 5 3` — the reverse. This is always true: mirroring a tree reverses its in-order traversal.

In code, it's just three lines inside the recursive function: swap `lc[node]` and `rc[node]` using a temporary variable, then recurse on both children.

## Path Sum: Passing a Running Total Down

This problem teaches another common tree pattern: carrying accumulated data *downward* through the recursion. You want to know if any root-to-leaf path sums to a target value.

The trick: instead of building the sum going up, **subtract the current node's value from the target going down.** When you reach a leaf, check if the remaining target equals the leaf's value. If it does, you've found a path.

Why subtract? Because it avoids needing to carry the sum *and* the target. You start with the target, and as you walk down, you "spend" each node's value. At a leaf, whatever's left must exactly equal the leaf's value for the path to work.

For the tree with values `10, 5, 15, 3, 7` and target 18:
- Start at 10, remaining = 18
- Go left to 5, remaining = 18 - 10 = 8
- Go left to 3, remaining = 8 - 5 = 3. Is 3 == 3 (leaf value)? **Yes!** Path: 10 → 5 → 3
