Memory bugs in C are notoriously hard to find. They often don't cause immediate crashes — instead, they corrupt data silently and cause failures later. Let's learn to recognize and avoid the most common ones.

## Memory Leaks

A memory leak occurs when you allocate memory but never free it:

<Editor defaultCode={`#include <stdio.h>\n#include <stdlib.h>\n\n// BAD: leaks memory\nvoid leak_example() {\n    int *arr = malloc(100 * sizeof(int));\n    // ... use arr ...\n    // Forgot to call free(arr)!\n    // When this function returns, the pointer is lost\n    // but the memory is still allocated.\n}\n\n// GOOD: properly frees memory\nvoid good_example() {\n    int *arr = malloc(100 * sizeof(int));\n    if (arr == NULL) return;\n    // ... use arr ...\n    free(arr);  // Always free when done\n}\n\nint main() {\n    // In a long-running program, leaks accumulate\n    // and can eventually exhaust all available memory\n    for (int i = 0; i < 5; i++) {\n        leak_example();  // Leaks 400 bytes each call\n    }\n    printf("Leaked some memory (you won't see an error)\\n");\n    \n    return 0;\n}\n`} time="O(1)" space="O(1)" />

<Callout type="warning">
Memory leaks don't crash your program immediately. They're slow poison — your program gradually consumes more and more memory until the system runs out. Tools like **Valgrind** (covered later) detect leaks automatically.
</Callout>

Common leak patterns:
- Forgetting to free before returning from a function
- Overwriting a pointer without freeing the old allocation
- Early returns that skip the `free` call

## Dangling Pointers

A dangling pointer points to memory that has already been freed:

```c
int *p = malloc(sizeof(int));
*p = 42;
free(p);

// p still holds the old address, but the memory is freed
printf("%d\n", *p);  // UNDEFINED BEHAVIOR
*p = 10;             // UNDEFINED BEHAVIOR
```

**Prevention**: Set pointers to NULL after freeing:
```c
free(p);
p = NULL;  // Now *p will segfault instead of silently corrupting data
```

## Use-After-Free

A specific case of dangling pointers — using memory after it's been freed:

<Editor defaultCode={`#include <stdio.h>\n#include <stdlib.h>\n\nint main() {\n    int *a = malloc(5 * sizeof(int));\n    if (a == NULL) return 1;\n    \n    for (int i = 0; i < 5; i++) a[i] = i * 10;\n    \n    printf("Before free: a[0] = %d\\n", a[0]);\n    \n    free(a);\n    a = NULL;  // Safety: prevents accidental use\n    \n    // If we hadn't set a to NULL, we might accidentally:\n    // printf("%d\\n", a[0]);  // Use-after-free!\n    \n    if (a == NULL) {\n        printf("Pointer is NULL — safe!\\n");\n    }\n    \n    return 0;\n}\n`} time="O(1)" space="O(1)" />

## Double Free

Freeing the same memory twice causes undefined behavior:

```c
int *p = malloc(sizeof(int));
free(p);
free(p);  // DOUBLE FREE — undefined behavior, potential crash
```

This can corrupt the memory allocator's internal data structures, leading to crashes or security vulnerabilities.

**Prevention**: Set to NULL after free — `free(NULL)` is safe and does nothing:
```c
free(p);
p = NULL;
free(p);  // Safe: free(NULL) is a no-op
```

## Buffer Overflow

Writing beyond the bounds of an allocated buffer:

```c
int *arr = malloc(5 * sizeof(int));

// Writing past the end
for (int i = 0; i <= 5; i++) {  // Bug: should be i < 5
    arr[i] = i;  // arr[5] is out of bounds!
}
```

Buffer overflows are the #1 source of security vulnerabilities in C. They can allow attackers to:
- Crash programs
- Execute arbitrary code
- Read sensitive data from memory

<Callout type="warning">
In 2014, the **Heartbleed** bug in OpenSSL was a buffer over-read that exposed private keys and passwords from millions of servers. It was caused by trusting a user-supplied length value without checking it against the actual buffer size.
</Callout>

## Uninitialized Pointers

Using a pointer before giving it a value:

```c
int *p;          // Uninitialized — points to random address
*p = 42;         // Writing to random memory — CRASH or corruption

int *q = NULL;   // Properly initialized — at least it's a known state
```

**Prevention**: Always initialize pointers — either to a valid address or to NULL.

## Checklist: Safe Memory Management

| Rule | Why |
|------|-----|
| Check `malloc` return for NULL | Allocation can fail |
| Always `free` what you `malloc` | Prevent memory leaks |
| Set pointers to NULL after `free` | Prevent use-after-free |
| Never `free` stack memory | Only free heap memory |
| Never `free` twice | Corrupts allocator |
| Never use memory after `free` | Data is gone |
| Check array bounds | Prevent buffer overflow |
| Initialize all pointers | Prevent wild pointer access |
