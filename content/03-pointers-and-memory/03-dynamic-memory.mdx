So far, every array you've declared has had a fixed size decided at compile time:
`int arr[100]`. But what if you're reading a file and don't know how many lines it
has? Or building a list that keeps growing? You need memory allocation at runtime.

This is what the **heap** is for.

## Stack vs Heap: Two Kinds of Memory

Your program uses two main regions for data:

**The stack** is like a stack of trays in a cafeteria. When you call a function, a new
tray (stack frame) goes on top with space for that function's local variables. When the
function returns, the tray is removed — all its variables vanish automatically. Fast,
but two limitations: the size is fixed at compile time, and it's relatively small
(typically 1-8 MB).

**The heap** is like a warehouse. You request a chunk of any size with `malloc`, use it
for as long as you need, then return it with `free`. No automatic cleanup — you're the
warehouse manager. If you forget to return a chunk, it stays reserved forever (a
*memory leak*). If you try to use a chunk after returning it, you're grabbing something
that may have been given to someone else (a *use-after-free* bug).

Why two regions? Because they serve different needs. The stack handles the common case
(small, short-lived data) with zero overhead. The heap handles the hard case
(unknown-size, long-lived data) with manual control.

## malloc — Memory Allocation

`malloc(size)` requests `size` bytes from the heap and returns a pointer to the start:

<Editor defaultCode={`#include <stdio.h>
#include <stdlib.h>

int main() {
    int n = 5;

    // Allocate space for 5 integers
    int *arr = malloc(n * sizeof(int));

    if (arr == NULL) {
        printf("Memory allocation failed!\\n");
        return 1;
    }

    // Use it like a normal array
    for (int i = 0; i < n; i++) {
        arr[i] = (i + 1) * 10;
    }

    for (int i = 0; i < n; i++) {
        printf("arr[%d] = %d\\n", i, arr[i]);
    }

    // Free when done
    free(arr);

    return 0;
}
`} time="O(n)" space="O(n)" />

What happens inside `malloc`? It asks the operating system (or its own internal free
list) for a contiguous block of the requested size. The OS returns a chunk of virtual
memory, and `malloc` gives you a pointer to it. The memory contains *garbage* — whatever
bytes were there before. It's your job to initialize it.

Why `sizeof(int)` and not just `4`? Because `int` isn't always 4 bytes on every
platform. `sizeof` gives the correct size on whatever machine you're compiling for.

## calloc — Cleared Allocation

`calloc(count, size)` does the same as `malloc` but **zeros all the bytes**:

<Editor defaultCode={`#include <stdio.h>
#include <stdlib.h>

int main() {
    int n = 5;

    // calloc: allocate and zero-initialize
    int *arr = calloc(n, sizeof(int));
    if (arr == NULL) return 1;

    // All values are guaranteed to be 0
    for (int i = 0; i < n; i++) {
        printf("arr[%d] = %d\\n", i, arr[i]);  // All zeros
    }

    free(arr);
    return 0;
}
`} time="O(n)" space="O(n)" />

Use `calloc` when you need a clean slate (arrays of counters, boolean flags, zeroed
buffers). Use `malloc` when you'll write every element anyway — it's slightly faster
since it skips the zeroing step.

## realloc — Resize Allocation

What if you allocated space for 100 elements but now need 200? `realloc` grows (or
shrinks) an existing allocation:

<Editor defaultCode={`#include <stdio.h>
#include <stdlib.h>

int main() {
    int capacity = 3;
    int *arr = malloc(capacity * sizeof(int));
    if (arr == NULL) return 1;

    arr[0] = 10;
    arr[1] = 20;
    arr[2] = 30;

    // Need more space — double the capacity
    capacity *= 2;
    int *temp = realloc(arr, capacity * sizeof(int));
    if (temp == NULL) {
        free(arr);  // Original still valid if realloc fails
        return 1;
    }
    arr = temp;

    arr[3] = 40;
    arr[4] = 50;
    arr[5] = 60;

    for (int i = 0; i < 6; i++) {
        printf("arr[%d] = %d\\n", i, arr[i]);
    }

    free(arr);
    return 0;
}
`} time="O(n)" space="O(n)" />

Under the hood, `realloc` tries to extend the block in place. If there's room after the
current block, it just adjusts the bookkeeping. If not, it allocates a new block,
copies everything over, and frees the old one. Either way, your existing data is
preserved.

<Callout type="warning">
Never write `arr = realloc(arr, new_size)`. If `realloc` fails, it returns NULL but
does NOT free the original. Assigning NULL to `arr` loses your only handle to that
memory — a guaranteed leak. Always use a temporary variable.
</Callout>

## free — Releasing Memory

Every `malloc`/`calloc`/`realloc` must have a matching `free`. Think of it as a
contract: you borrowed memory from the OS, and `free` returns it.

```c
int *p = malloc(100 * sizeof(int));
// ... use p ...
free(p);     // Release the memory
p = NULL;    // Good practice: prevent use-after-free
```

Setting the pointer to NULL after freeing is a safety habit. Dereferencing NULL
causes a clean crash (segfault), while dereferencing freed memory causes
*unpredictable* corruption. Clean crashes are much easier to debug.

## The Ownership Question

Dynamic memory introduces a problem that doesn't exist with stack variables: **who
is responsible for freeing this memory?**

If `create_array()` calls `malloc` and returns the pointer, the *caller* must call
`free`. If the caller passes that pointer to three other functions, which one frees
it? If no one does, it's a leak. If two of them do, it's a double-free crash.

This is the **ownership** problem. C gives you no language-level tools to manage it —
you must establish conventions. The simplest: whoever `malloc`s is responsible for
`free`ing. When you can't follow that (like factory functions), document it clearly:
"Caller must free the returned pointer."

## A Dynamic Array Example

Putting it together — a growable array that doubles its capacity when full:

<Editor defaultCode={`#include <stdio.h>
#include <stdlib.h>

int main() {
    int capacity = 4;
    int size = 0;
    int *arr = malloc(capacity * sizeof(int));
    if (arr == NULL) return 1;

    // Read numbers until we see -1
    int value;
    scanf("%d", &value);
    while (value != -1) {
        // Grow if needed
        if (size == capacity) {
            capacity *= 2;
            int *temp = realloc(arr, capacity * sizeof(int));
            if (temp == NULL) { free(arr); return 1; }
            arr = temp;
        }

        arr[size++] = value;
        scanf("%d", &value);
    }

    printf("Read %d values:\\n", size);
    for (int i = 0; i < size; i++) {
        printf("%d ", arr[i]);
    }
    printf("\\n");

    free(arr);
    return 0;
}
`} stdin="10 20 30 40 50 60 70 80 -1" time="O(n)" space="O(n)" />

Why double the capacity instead of adding 1? Because `realloc` may copy the entire
array to a new location. If you grow by 1 each time and add n elements, that's
1 + 2 + 3 + ... + n = O(n^2) copies total. Doubling means at most O(n) total copies
across all reallocs — this is called **amortized O(1)** per insertion.
