Every C programmer eventually hits a bug that `printf` can't solve. Maybe the program crashes with no output at all. Maybe it corrupts memory ten lines before the actual crash. Maybe an off-by-one error only shows up when the input has exactly 1,000 items and you can't figure out which iteration goes wrong.

This is where a **debugger** comes in. GDB (the GNU Debugger) gives you something `printf` never can: the ability to **freeze a running program in time**, examine exactly what's happening at any point in execution, and step through your code one line at a time. Think of it as a time machine for your program — you can set "bookmarks" (breakpoints) at interesting moments, travel to them, look around at every variable and call, then continue forward at your own pace.

## Compiling for Debugging

Before GDB can help you, it needs a map. When the compiler turns your C code into machine instructions, it normally throws away all the names — variable names, function names, line numbers. The `-g` flag tells the compiler to keep this information, embedding a lookup table called **debug symbols** into the binary. These symbols are the map from raw addresses back to your source code.

```
gcc -g -o program program.c
```

Without `-g`, GDB can show you raw assembly and memory addresses, but not the source lines or human-readable names. It's like trying to navigate a city with a street map versus GPS coordinates — technically possible, but painful.

<Callout type="tip">
Also add `-O0` to disable optimizations. When the compiler optimizes, it reorders, combines, and eliminates statements. A variable you declared might not exist in the optimized code. A loop might be unrolled. Stepping through optimized code in GDB feels like reading a book with the pages shuffled: `gcc -g -O0 -o program program.c`
</Callout>

## Starting GDB

```
gdb ./program          # Start debugging
gdb --args ./prog a b  # Pass command-line arguments
```

Once inside GDB, you're in a conversation with your program. You tell it where to pause, when to proceed, and what to show you. Here are the core commands — you'll use these constantly:

// Compile: gcc -g -O0 -o demo demo.c
// Run:     gdb ./demo

// GDB Command Reference:
//

//   quit (q)             - Exit GDB
//

//   break main           - Break at function

//   info breakpoints     - List all breakpoints
//   delete 1             - Delete breakpoint #1
//   disable 2            - Temporarily disable #2
//
// STEPPING:
//   next (n)             - Execute next line (step OVER functions)
//   step (s)             - Execute next line (step INTO functions)

//
// INSPECTING:
//   print x              - Print variable x

    return n * factorial(n - 1);
}

int main() {

    int result = factorial(5);
    printf("5! = %d\\n", result);

The key mental model is the difference between `next` and `step`. Imagine you're reading a recipe. `next` says "do the next instruction and tell me the result" — if it says "make the sauce," you get the finished sauce. `step` says "show me the sub-recipe for making the sauce" — you dive into the function. Use `next` when you trust a function works correctly; use `step` when you suspect the bug is inside it.

## Debugging a Crash

When your program segfaults, most output is gone. `printf` debugging is useless because the crash might happen before the output buffer flushes. But GDB catches the crash in the act. It freezes the program at the exact instruction that caused the fault, and you can use `backtrace` to see the entire chain of function calls that led there.

This is the standard crash debugging workflow:

1. `gdb ./program` — load the program
2. `run` — let it execute until it crashes  
3. `bt` (backtrace) — see WHERE it crashed and the call chain
4. `frame N` — jump to any frame in the chain to examine context
5. `info locals` — see what variables looked like at that point
6. `print expr` — evaluate any expression in that frame's context

## Conditional Breakpoints

When you have a bug that only appears on the 99th iteration of a loop, you don't want to hit `next` 99 times. Conditional breakpoints let you tell GDB: "only stop here when this expression is true." This is the single biggest time-saver over `printf` debugging — with `printf`, you'd need to add a conditional print, recompile, and re-run. With GDB, you just type:

```
break 25 if i == 99        # Break at line 25 only when i is 99
break func if x > 100      # Break in func when x exceeds 100
```

## Why Learn GDB Even If You Have an IDE

Modern IDEs have graphical debuggers with "click to add breakpoint" and variable inspection panels. They're great for everyday use. But GDB gives you capabilities that no GUI can:

- **Remote debugging**: Your program runs on a server or embedded device, and you debug it from your laptop over SSH. This is the only option for headless servers and production crashes.
- **Core dump analysis**: When a program crashes in production, the OS can save a snapshot of its memory (a "core dump"). GDB can open this snapshot and let you inspect the state at the moment of the crash — days or weeks later, on a different machine.
- **Scripting**: GDB has a built-in scripting language. You can automate complex debugging workflows, like "every time function X is called, print parameter Y and continue."

Learning GDB is like learning the command line versus a file manager — more effort up front, but vastly more powerful in the situations that matter most.
