Graphs are arguably the most versatile data structure in computer science. Any time you have *things* and *relationships between things*, you have a graph — social networks (people and friendships), maps (intersections and roads), the internet (pages and links), build systems (tasks and dependencies), even the neurons in your brain.

Unlike trees, graphs have no root, no strict hierarchy, and can contain cycles. That freedom is what makes them powerful — and what makes them tricky to work with.

## What Is a Graph?

A graph consists of **vertices** (nodes) and **edges** (connections):

```
A --- B       Undirected graph
|   / |       Vertices: {A, B, C, D}
|  /  |       Edges: {(A,B), (A,C), (B,C), (B,D)}
| /   |
C --- D
```

The first question you face when implementing a graph is: *how do you store it?* There are two standard answers, and the choice between them matters a lot.

## Graph Representations: A Critical Tradeoff

### Adjacency Matrix

An adjacency matrix uses a 2D array where `adj[i][j] = 1` means there's an edge between vertex `i` and vertex `j`. It's the most intuitive representation: a grid where you can check any connection in O(1).

**When to use it**: dense graphs where most vertices are connected to most other vertices. In a complete graph with `V` vertices, you have `V*(V-1)/2` edges — the matrix is mostly full, so the O(V^2) space isn't wasteful.

**The problem**: for sparse graphs (which are *most* real-world graphs), an adjacency matrix wastes enormous space. A social network might have a billion users but each person has only a few hundred friends. That's a billion-by-billion matrix (10^18 entries) to store a few hundred billion edges. The matrix would be 99.99999% zeros.

Also, finding all neighbors of a vertex requires scanning an entire row — O(V) — even if the vertex only has 3 neighbors.

<Editor defaultCode={`#include <stdio.h>
#include <string.h>

#define V 5

int main() {
    // 0=A, 1=B, 2=C, 3=D, 4=E
    // Edges: A-B, A-C, B-C, B-D, C-D, D-E
    int adj[V][V] = {0};

    // Add undirected edges
    adj[0][1] = adj[1][0] = 1;  // A-B
    adj[0][2] = adj[2][0] = 1;  // A-C
    adj[1][2] = adj[2][1] = 1;  // B-C
    adj[1][3] = adj[3][1] = 1;  // B-D
    adj[2][3] = adj[3][2] = 1;  // C-D
    adj[3][4] = adj[4][3] = 1;  // D-E

    char *names[] = {"A", "B", "C", "D", "E"};

    printf("Adjacency Matrix:\\n  ");
    for (int i = 0; i < V; i++) printf("%s ", names[i]);
    printf("\\n");

    for (int i = 0; i < V; i++) {
        printf("%s ", names[i]);
        for (int j = 0; j < V; j++) {
            printf("%d ", adj[i][j]);
        }
        printf("\\n");
    }

    // Print neighbors
    printf("\\nNeighbors of B: ");
    for (int j = 0; j < V; j++) {
        if (adj[1][j]) printf("%s ", names[j]);
    }
    printf("\\n");

    return 0;
}
`} time="O(V²)" space="O(V²)" />

### Adjacency List

An adjacency list stores, for each vertex, only the vertices it's actually connected to. For a vertex with 3 neighbors, you store exactly 3 entries — not `V` entries with `V-3` zeros.

**When to use it**: sparse graphs, which is almost always. Total space is O(V + E), which for a sparse graph is dramatically smaller than O(V^2).

The tradeoff: checking whether a specific edge exists is no longer O(1). You have to walk the linked list for that vertex, which is O(degree). But for most graph algorithms (BFS, DFS, Dijkstra's), you iterate over all neighbors anyway — and adjacency lists do that in O(degree) while matrices waste O(V) scanning zeros.

| Operation | Adjacency Matrix | Adjacency List |
|---|---|---|
| Check if edge exists | O(1) | O(degree) |
| List all neighbors | O(V) | O(degree) |
| Add an edge | O(1) | O(1) |
| Space | O(V^2) | O(V + E) |

For most practical applications — especially sparse graphs with thousands or millions of vertices — **adjacency lists win**.

<Editor defaultCode={`#include <stdio.h>
#include <stdlib.h>

#define V 5

typedef struct Node {
    int vertex;
    struct Node *next;
} Node;

Node *adj[V] = {NULL};

void add_edge(int u, int v) {
    Node *node = malloc(sizeof(Node));
    node->vertex = v;
    node->next = adj[u];
    adj[u] = node;

    node = malloc(sizeof(Node));
    node->vertex = u;
    node->next = adj[v];
    adj[v] = node;
}

int main() {
    char *names[] = {"A", "B", "C", "D", "E"};

    add_edge(0, 1);  // A-B
    add_edge(0, 2);  // A-C
    add_edge(1, 2);  // B-C
    add_edge(1, 3);  // B-D
    add_edge(2, 3);  // C-D
    add_edge(3, 4);  // D-E

    printf("Adjacency List:\\n");
    for (int i = 0; i < V; i++) {
        printf("%s: ", names[i]);
        for (Node *n = adj[i]; n; n = n->next) {
            printf("%s ", names[n->vertex]);
        }
        printf("\\n");
    }

    // Free memory
    for (int i = 0; i < V; i++) {
        Node *n = adj[i];
        while (n) {
            Node *next = n->next;
            free(n);
            n = next;
        }
    }

    return 0;
}
`} time="O(V + E)" space="O(V + E)" />

## BFS vs DFS: Two Ways to Explore

Graph traversal is about systematically visiting every reachable vertex. There are two fundamentally different strategies, and understanding *when* to use each is one of the most important skills in algorithm design.

### The Mental Model: Explore vs Expand

Think of exploring a maze:

**DFS (Depth-First Search)** is like *exploring*: pick a corridor and follow it as far as you can. Hit a dead end? Backtrack and try the next unexplored branch. You go *deep* before going *wide*. DFS uses a **stack** (or recursion, which uses the call stack implicitly).

**BFS (Breadth-First Search)** is like *expanding*: stand at the entrance and look at every corridor one step away. Then look at every corridor two steps away. You go *wide* before going *deep*. BFS uses a **queue**.

### When to Use Which

**Use BFS when you need shortest paths** in an unweighted graph. Because BFS expands level-by-level, the first time it reaches any vertex is guaranteed to be via the shortest path. This is probably the single most important fact about BFS.

**Use DFS when you need to explore structure**: cycle detection, connected components, topological sorting, finding strongly connected components. DFS is also natural for problems with backtracking (like solving a maze or generating permutations).

A practical mnemonic: BFS answers "how far?" and DFS answers "is there a path?" (both can do both, but each is natural for one).

## Breadth-First Search (BFS)

BFS explores level by level. It guarantees that when you first reach a vertex, the distance recorded is the shortest possible. This is because it processes all vertices at distance `d` before any vertex at distance `d+1`.

<Editor defaultCode={`#include <stdio.h>
#include <string.h>

#define V 6

int adj[V][V] = {0};

void bfs(int start) {
    int visited[V] = {0};
    int queue[V], front = 0, back = 0;
    int distance[V];
    memset(distance, -1, sizeof(distance));

    visited[start] = 1;
    distance[start] = 0;
    queue[back++] = start;

    while (front < back) {
        int u = queue[front++];
        printf("Visit %d (distance %d)\\n", u, distance[u]);

        for (int v = 0; v < V; v++) {
            if (adj[u][v] && !visited[v]) {
                visited[v] = 1;
                distance[v] = distance[u] + 1;
                queue[back++] = v;
            }
        }
    }
}

int main() {
    // 0-1, 0-2, 1-3, 2-3, 2-4, 3-5
    adj[0][1]=adj[1][0]=1; adj[0][2]=adj[2][0]=1;
    adj[1][3]=adj[3][1]=1; adj[2][3]=adj[3][2]=1;
    adj[2][4]=adj[4][2]=1; adj[3][5]=adj[5][3]=1;

    printf("BFS from vertex 0:\\n");
    bfs(0);

    return 0;
}
`} time="O(V + E)" space="O(V)" />

## Depth-First Search (DFS)

DFS plunges as deep as it can along each branch before backtracking. It's naturally recursive — the call stack acts as the "backtracking memory." DFS is the workhorse for structural graph problems: detecting cycles (if you encounter an already-visited vertex during traversal, there's a cycle), counting connected components (each fresh DFS call discovers one component), and topological sorting.

<Editor defaultCode={`#include <stdio.h>

#define V 6

int adj[V][V] = {0};
int visited[V] = {0};

void dfs(int u) {
    visited[u] = 1;
    printf("Visit %d\\n", u);

    for (int v = 0; v < V; v++) {
        if (adj[u][v] && !visited[v]) {
            dfs(v);
        }
    }
}

int main() {
    adj[0][1]=adj[1][0]=1; adj[0][2]=adj[2][0]=1;
    adj[1][3]=adj[3][1]=1; adj[2][3]=adj[3][2]=1;
    adj[2][4]=adj[4][2]=1; adj[3][5]=adj[5][3]=1;

    printf("DFS from vertex 0:\\n");
    dfs(0);

    return 0;
}
`} time="O(V + E)" space="O(V)" />

## Topological Sort: Ordering Dependencies

Topological sort answers the question: "In what order should I do these tasks, given that some depend on others?" Think of course prerequisites — you can't take Data Structures before Intro to Programming. Or think of build systems: `main.c` depends on `utils.h`, which depends on `config.h`. You need an order where every dependency is satisfied before the thing that depends on it.

A topological ordering exists **only on Directed Acyclic Graphs (DAGs)** — directed graphs with no cycles. The reason is intuitive: if A depends on B and B depends on A, there's no valid ordering. A cycle means mutual dependency, which is a deadlock.

The DFS-based approach works by running DFS and pushing each vertex onto a stack *after* all its descendants are fully explored. The reversed result is a valid topological order — because a vertex only appears after everything it depends on.

<Callout type="info">
Topological sort fails on graphs with cycles — if you detect a cycle during DFS (a "back edge" to an already-in-progress vertex), the graph has no valid topological order. This is actually a useful cycle-detection technique in directed graphs.
</Callout>

#include <string.h>

#define MAX_V 100

int adj[MAX_V][MAX_V];
int visited[MAX_V];

void dfs(int u, int v_count) {
    visited[u] = 1;
    for (int v = 0; v < v_count; v++) {
        if (adj[u][v] && !visited[v]) {
            dfs(v, v_count);
        }
    }
}

int main() {
    int v, e;
    scanf("%d %d", &v, &e);

    memset(adj, 0, sizeof(adj));
    memset(visited, 0, sizeof(visited));

    for (int i = 0; i < e; i++) {
        int a, b;
        scanf("%d %d", &a, &b);
        adj[a][b] = adj[b][a] = 1;
    }

    int components = 0;
    for (int i = 0; i < v; i++) {
        if (!visited[i]) {
            dfs(i, v);
            components++;
        }
    }

    printf("%d\\n", components);
    return 0;
}

## Exercise: Shortest Path (BFS)

<Exercise
  title="Unweighted shortest path"
  description="Read V, E, and edges. Then read source S and destination D. Print the shortest distance (number of edges) from S to D, or -1 if unreachable."
  starterCode={`#include <stdio.h>
#include <string.h>

#define MAX_V 100

int adj[MAX_V][MAX_V];

int main() {
    int v, e;
    scanf("%d %d", &v, &e);

    memset(adj, 0, sizeof(adj));

    for (int i = 0; i < e; i++) {
        int a, b;
        scanf("%d %d", &a, &b);
        adj[a][b] = adj[b][a] = 1;
    }

    int s, d;
    scanf("%d %d", &s, &d);

    // BFS
    int dist[MAX_V];
    memset(dist, -1, sizeof(dist));
    int queue[MAX_V], front = 0, back = 0;

    dist[s] = 0;
    queue[back++] = s;

    while (front < back) {
        int u = queue[front++];
        if (u == d) break;
        for (int i = 0; i < v; i++) {
            if (adj[u][i] && dist[i] == -1) {
                dist[i] = dist[u] + 1;
                queue[back++] = i;
            }
        }
    }

    printf("%d\\n", dist[d]);
    return 0;
}
